QMO-to-CMM Mapping
Overview
The Capability Maturity Model for Software (CMM) developed by the Software Engineering Institute (SEI) of Carnegie Mellon University is a framework that describes the key elements of an effective software process. CMM is the most sought after standard in the US and is rapidly expanding internationally.

The CMM describes an evolutionary improvement path for software organizations from an ad hoc, immature process to a mature, disciplined one. This path is encompassed by five levels of maturity. They are:

  1. Initial - The software process is characterized as ad hoc, and occasionally even chaotic. Few processes are defined, and success depends on individual effort and heroics.
  2. Repeatable - Basic project management processes are established to track cost, schedule, and functionality. The necessary process discipline is in place to repeat earlier successes on projects with similar applications.
  3. Defined - The software process for both management and engineering activities is documented, standardized, and integrated into a standard software process for the organization. All projects use an approved, tailored version of the organization's standard software process for developing and maintaining software.
  4. Managed - Detailed measures of the software process and product quality are collected. Both the software process and products are quantitatively understood and controlled.
  5. Optimizing - Continuous process improvement is enabled by quantitative feedback from the process and from piloting innovative ideas and technologies.
SEI is currently updating the CMM. Version 2.0 is expected to be available by mid-1999. Levels 4 and 5 will be heavily impacted.

QMO-to-CMM Mapping
LEVEL 2 - REPEATABLE
Requirements Management
  1. Requirements allocated to software are controlled to establish a baseline for software engineering and management use.
    • Development of Requirements
    • Baseline of Requirements (internal and external)
  2. Software plans, products, and activities are kept consistent with the requirement allocated to software.
    • Baselined Project Plan
    • Baselined Requirements
    • Baselined Designs
    • Baselined Test Case Specifications
    • Change Control Procedure
    • Baselined Technical Operating Environment
    • Baselined System Architecture
    • Traceability
Project Planning 
  1. Software estimates are documented for use in planning and tracking the software project.
    • Estimation summary sheet
    • Thin Requirements
    • Pre-Commitment Project Plan
    • Project Plans and their project schedules
    • Lessons Learned at project close-out
  2. Software activities and commitments are planned and documented.
    • Baselined Project Plan
    • Baselined Configuration Management Plan
    • Baselined Technical Operating Environment
    • Pre-Commitment Architecture Specifications
    • Baselined Master Test Plan
    • Information Development Plan
  3. Affected groups and individuals agree to their commitments related to the software project.
    • Risk Management (Documented in Project Plan and reviewed at Project Status Meetings)
    • Baselined Requirements, Designs, Technical Operating Environment, System Architecture, Test Case Specifications, and Master Test Plan: e.g., Review Tracking Sheets
Project Tracking and Oversight
  1. Actual results and performance are tracked against the software plans.
    • Project Status Meeting: schedule, budget/cost, resources, risk(s), jeopardy(s), action item register, and QUIP
    • Blueprint
  2. Corrective actions are taken and managed to closure when actual results and performance deviate significantly from software plans
    • Change Control Procedure
    • Jeopardy Process
    • Project Status Meeting, Schedule, Action Item Register
  3. Changes to software commitments are agreed to by all affected groups and individuals.
    • Change Control Procedure/Review Tracking Sheet
Subcontract Management
  1. The prime contractor selects qualified software subcontractors.
    • Subcontractor Management
  2. The prime contractor and the software subcontractor agree to their commitments to each other.
    • Subcontractor Management
  3. The prime contractor and the software subcontractor maintain communications.
    • Subcontractor Management
  4. The prime contractor tracks the software subcontractor's actual results and performance against its commitments.
    • Subcontractor Management
Software Quality Assurance
  1. Software quality assurance activities are planned.
    • Internal Audit Schedule
    • QUIP
    • Master Test Plan
    • Lessons Learned
    • Root Cause Analysis
  2. Adherence of products and activities to the applicable standards, procedures, and requirements is verified objectively.
    • Internal Audit Process
    • Baseline Process/Review Tracking Sheet(s) for baselined documents
    • Project self-audits
    • Test Case Specifications
  3. Affected groups and individuals are informed of software quality assurance activities and results.
    • Project-level QMT meetings (documented Internal Audit results)
    • Internal Audit schedules, and closing conferences
    • Test Metrics
  4. Noncompliance issues that cannot be resolved within the software project are addressed by senior management.
    • Internal Audit escalation procedure
    • Jeopardy Process
    • QMT meetings
Configuration Management
  1. Configuration management activities are planned.
    • Baselined Configuration Management Plan
    • Build Schedule
  2. Selected software work products are identified, controlled, and available.
    • Baselined Configuration Management Plan
    • Build Schedule
    • Trouble Tracking Tool
  3. Changes to identified work products are controlled.
    • Change Control Procedure
    • Change Control Log/Review Tracking Sheet (including CR process)
  4. Affected groups and individuals are informed of the status and content of software baselines.
    • Integrity Report
    • Change Control Log/Review Tracking Sheet
LEVEL 3 - DEFINED
Organization Process Focus
  1. Software process development and improvement activities are coordinated across the organization.
    • QMO Release Process
    • QMT
    • Management Review (QMT)
    • Project QUIP
    • Q-Tip
  2. The strengths and weaknesses of the software processes used are identified relative to a process standard.
    • CMM assessment
    • Internal Audits
  3. Organization-level process development and improvement activities are planned.
    • QMO Release Process
    • QMT
    • QUIP
    • Project QUIP
    • QMO Training
Organization Process Definition
  1. A standard software process for the organization is developed and maintained.
    • QMO
    • Local procedures
  2. Information related to the use of the organization's standard software process by the software projects is collected, reviewed, and made available.
    • Project Library
    • Metrics
    • QMT
    • Q-Tips
Integrated Software Management
  1. The project's defined software process is a tailored version of the organization's standard software process.
    • QMO
    • Local procedures
    • Job Aid
  2. The project is planned and managed according to the project's defined software process.
    • QMO
    • Local Procedures
    • QMO Exception Process
    • Schedule
    • Jeopardy Process
    • Project Status Meeting
Software Product Engineering
  1. The software engineering tasks are defined, integrated, and consistently performed to produce the software.
    • QMO
    • Project Teams
    • Local Procedures
    • Local test Practices
    • Information Development Plan
  2. Software work products are kept consistent with each other.
    • Change Control Procedure/Review Tracking Sheet
    • Traceability
    Training Program
    1. Training activities are planned.
      • Training Plans
    2. Training for developing the skills and knowledge needed to perform software management and technical roles is provided.
      • Training Plans
    3. Individuals in the software engineering group and software-related groups receive the training necessary to perform their roles.
      • Training Plans and Training Records
      Intergroup Coordination
      1. The client's requirements are agreed to by all affected groups.
        • Baselined Requirements
        • Review Tracking Sheet
        • Client concurrence
        • Project Teams
      2. The commitments between the engineering groups are agreed to by the affected groups.
        • Baseline Process
        • Project Teams
        • Project Status Meeting / Action Item Register
        • QMT
      3. The engineering groups identify, track, and resolve intergroup issues.
        • Project Teams
        • Project Status Meeting / Action Item Register
        • Jeopardy
        • Review Tracking Sheet
        Peer Reviews
        1. Peer review activities are planned.
          • Baseline Process
          • Review Process
          • Code Inspections
        2. Defects in the software work products are identified and removed.
          • Baseline Process
          • Review Process
          • Code Inspections
          • Trouble Tracking Tool
