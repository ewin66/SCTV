4.5 Develop Code and Conduct Unit Test
Purpose Code is developed to implement the software features as well as the installation/transition process described in the Detailed Design. Conventional coding practices and applicable project level practices should be used in programming software from the Detailed Design documents. 

Projects should follow the U S WEST development standards. 
Entrance Criteria  Baselined System Architecture
Baselined TOE
Baselined Requirements
Baselined Test Plan
Baselined Detailed Design
Draft Test Case Specifications 
Steps 
  1. Create/update source files (Development). Source files include source code, header files, make files, input scripts, input tables, alarm and recovery tables, etc. If using prototype code in the final delivery, update to production quality. See technique below for a Job Aid. 
  2. Write installation/transition procedures (Development). 
  3. Write conversion routine, if applicable (Development). Include common and reusable code in the system. Consider impacts to reusable code and interfacing modules. 
  4. Meet all criteria defined in the Requirements. 
  5. Compile and link (Development). 
  6. Perform Unit Test Planning 
    • Plan the general approach, resources and schedules 
    • Determine the breakdown of unit test (e.g., feature, modules, set of modules) Plan 
    • Refine the Master Test Plan with information from the Unit Test Plan 
    Acquire the Unit Test Set 
    • Design the set of unit tests (i.e., design inputs, tasks, outputs) 
    • Implement the refined Unit Test Plan and design 
    Measure the Test Unit 
    • Run unit tests 
    • Determine results 
      • Record pass/fail 
      • For each failure, have the failure analyzed and recorded 
    Provide unit test status to Project Manager 
    Developer must sign-off unit test results, and retain documentation throughout the integration test phase 
  7. Place files under source code control. See Baseline Process, Change Management and Configuration Management for details. 
  8. Write Runbooks/Man Pages, if applicable (Development). 
  9. Deliver the system to Integration Test for acceptance (Development or Change Management Organization). 
  10. Update the Blueprint (Project Manager).

Meets integration test acceptance criteria

4.6 Conduct Code Inspection
Purpose  To ensure that the design document, the Requirements and the code meet project coding standards (e.g., Y2K functional standards, C and C++ Coding Standards, IT Coding Standards or local refinements [TPSP]).

 
Entrance Criteria  One or more software units to be code inspected

 
Steps 
  1. Use of the following code inspection plan/strategy is recommended: 
    • New Code: 100% inspected 
    • Changed Code: All modules changed must be inspected at least once 
    • Reinspection of a module will be based on the judgment of the development team 
  2. Conduct formal or informal code inspections (Development). 

    3.  

     
     
     

    The Developer performs the following activities: 

    • scheduling the code inspections --- as soon as possible after the first clean compile of the code 
    • determining if inspection procedure will be formal or informal 
    • preparing materials for the code inspection; Inspection team members are expected to review these materials prior to the meeting 
    • conducting a code inspection meeting (to locate errors in the software, not to propose solutions) 
    • completing a code inspection log 
    • performing post-inspection activities --- such as completing the Code Inspection Summary Form, and updating metrics 
    Formal: 
    • identify inspection team - Author, moderator, reader, recorder and inspectors (i.e., specific roles assigned to participants) 
    • usually performed in a meeting format 
    • materials are distributed and reviewed prior to meeting 
    • uses industry recognized methods such as Fagan or Cleanroom 
    Informal: 
    • identify one or more inspector(s) 
    • usually not performed in a meeting format 
    • materials are distributed and reviewed prior to the inspection 
    • inspection method is less formal, however, the goal is to locate discrepancies between the Requirements and the code 
  3. Maintain code inspection metrics summarizing data by release 
  4. Reinspect module after faults have been corrected (if complexity of the changes or the number of faults warrant a reinspection). Resolve action items (Development). 
  5. Correct faults identified in the inspection. 
  6. Retain the metrics in the Project Library for easy retrieval (Project Manager). 
Exit Criteria  Code inspection metrics 
No reinspection necessary (date entered on Blueprint)
Updated Blueprint
Quality Records Code inspection metrics 
Updated Blueprint
Code Inspection Log
Code Inspection Summary 
Controlled Documents None 
Metrics Code Inspection Metrics summarizing data by release
Updates to or actuals for size and cost 
Templates  Code Inspection Log Template
Code Inspection Summary Template
Tools As Appropriate 
Techniques The following are valid implementations of Code Inspection: 
  1. All new and changed code, including CRs that are interface-type code: inspect interfacing modules in a formal setting with the test cases previously defined (available at review). This is most effective if all interfaces are present. 
  2. All new and changed code, including CRs that are complex algorithmic code: inspection conducted using a formal procedure with test cases previously defined (available at review). 
  3. All new and changed code that is going to be "commonly used by others": inspection conducted using a formal procedure with test cases previously defined (available at review). 
  4. Code to correct a CR that is contained within a module that has been "known to contain other problems": inspection conducted using a formal procedure. 
  5. Code to correct a CR that is interface-type code: inspecting interfacing modules in a formal setting with the test cases previously defined (available at review) OR less formal depending on the complexity of the fix. 
  6. Code to correct a CR (that is not interfacing or complex): informal procedure.

4.10 Conduct Integration Test
Purpose  Integration Test is conducted to ensure the integrity of an integrated software system or subsystem when exercising internal interfaces.

 
Entrance Criteria  Baselined Test Case Specifications
Code Inspection complete
Unit Test complete
Information Development Materials (e.g., Installation and Transition Documents, On-Line Materials) (draft) 
Steps 
  1. Appoint a person (may be Integration Test team leader) to monitor Integration Test and approve Integration Test Sign-Off Criteria Sheet or the Integration Test Exit Criteria defined in Test Plan (Integration Test Manager). 
  2. Execute Integration Test acceptance tests as defined in the Test Case Specification: (Development or Test) 
    • Review actual and expected results 
    • Determine if the system is ready for Integration Test 
      • If it is not ready, document discrepancy using a problem tracking system; await re-delivery to Integration Test. 
  3. Execute Integration Test cases defined in the Test Case Specification: (Development or Test) 
    • Review actual and expected results; there should be no discrepancies 
    • Document any test discrepancies using a problem tracking system 
    • Verify that fixes are mapped to a discrepancy report (use a trouble tracking system) 
    • Test the Information Development materials focusing on the technical accuracy of new information that specifically supports new features and CRs. Local Checklists may be used in lieu of the Review Tracking Sheets during Integration Test 
    • Retain or reproduce successful (passed) test results until the stated U S WEST exit criteria has been met. 

      • The actual Integration Test test results are transient quality records. 
    • Perform regression tests. 
  4. Maintain Integration Test metrics. (weekly) to demonstrate pass/fail testing (Development or Test). The weekly Integration Test metrics are transient quality records; that is, the new metrics will supersede the old. Maintain the final week's summary result of Integration Test Metrics as the quality record after successfully passing Integration Test. 
  5. Verify that the Integration Test Exit Criteria defined in the Master Test Plan was met and approve Integration Test Sign-off Criteria Sheet (Person in Step 1). 
  6. Deliver the system to System Test for acceptance (Integration Test). 
  7. Retain the appropriate artifacts in a Project Library for easy retrieval (Project Manager). 
Exit Criteria  Approved Integration Test Sign-off Criteria Sheet (met Integration Test Exit Criteria defined in Master Test Plan) (Q-7)
Integration Test test metrics (final week of Integration Test only) (Q-7)
Integration Test tested Information Development deliverables as outlined in the Information Development Release Plan (Q-7)
Project can be installed/transitioned using delivered installation/transition software and documentation (Q-7)
Quality Records  Approved Integration Test Sign-Off Criteria Sheet (met Integration Test Exit Criteria defined in Master Test Plan) (Q-7)
Integration Test metrics (final week of Integration Test)
Final pass/fail status of each Integration Test test case
Defects found during Integration Test (from the problem-tracking tool)
Review Tracking Sheets or QA (Quality Assurance) Checklists for Information Development documents (Disposition codes of 1 or 2 should be interpreted to mean "Ready for System Test" and a code of 3 as "Not Ready")

 
Controlled Documents None 
Metrics Updates to or actuals for size and cost

4.11 Conduct Development Lessons Learned Session 
Purpose  A Lessons Learned Session is conducted after Development is completed to evaluate the effectiveness of, and document improvements to, the Development process.

 This session is optional and may be initiated by any person involved in the phase. The person who initiates the Lessons Learned Session is responsible for appointing a facilitator.

 
Entrance Criteria  Integration Test Completed 
Steps 
  1. Determine the participants. Schedule and announce the Lessons Learned Session (Initiator).
  2. Use brainstorming or other techniques to determine processes that were especially effective or could be improved (Facilitator).
  3. Write meeting minutes that include: (Facilitator) 
    • action items 
    • person responsible for action items 
    • due date for action items 
  4. Provide information to Project Manager to update the Quality Improvement Plan (if applicable). See Quality Improvement Plan (QUIP) (found in Appendix) for details. (Initiator)
Individual project teams may further refine these steps as necessary.

 
Exit Criteria  Meeting minutes and/or Action Item Register 
Quality Records  Updated QUIP
Meeting minutes and/or completed Action Item Register 
Controlled Documents None 
Metrics None 
Templates  None
