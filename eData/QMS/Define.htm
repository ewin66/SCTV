2.2 Develop Draft Requirements
Purpose  Requirements are developed to capture clients' needs and provide a benchmark to which designs will be validated. Requirements should: 
  • specify what the system does without specifying how it should be done 
  • be precise 
  • include performance, security, reliability, portability and other constraints 
  • define inter-requirements relationships 
An effective and recommended technique for capturing requirements is to conduct a series of interactive interview sessions with the client to fully understand the business problem. 

There are several approaches to this activity: by feature/CR, iteratively, incrementally, or by waterfall. Listed at the end of this section are tools and techniques to facilitate information generation (re-use) as well as other time and cost saving implementation alternatives. 
Entrance
Criteria 		 Baselined Thin Requirements 
Baselined Project Plan
Steps  These steps may overlap client involvement throughout the process.
  1. Create conceptual/logical models (e.g., 1: data: entities and relationships and 2: process: data flow diagrams) (Analyst).
  2. Define the new or modified data elements, for example: (Analyst)
    • data item type
    • length
    • format
    • description
    • verification, validation and deviation rules 
  3. Identify system functions, e.g., batch, on-line, triggering events, processing, and multiple event transactions, use cases (Analyst).
  4. Specify data requirements (Analyst).
  5. Specify subsystems that: (Analyst)
    • determine technology, performance, and security constraints
    • identify reusable designs
    • review the SOW or generic requirements to reflect information obtained in the Client Interface design
    • create a preliminary or use an existing subsystem design for each subsystem
    • specify how the appropriate projects are used in this system
    • specify subsystem boundaries
    • specify external interfaces
    • specify automated/manual processes
    • specify mode:
      • interactive/batch
      • media (e.g., tape, on-line)
      • how often used
  6. Specify security and control requirements, by defining: (Analyst)
    • the points for application controls to ensure complete, accurate, and authorized processing of information
    • the points and criteria to consider if fraudulent use of the system
    • the security requirements for data stores, including access authorizations and any additional security and control requirements
  7. Specify backup requirements, with these considerations: (Analyst)
    • Machine --- identify preventive or contingency plans based on the possibility of system failure (partial or complete) or extended downtime
    • Data --- identify the backup requirements for databases and data stores
  8. Establish error handling requirements: (Analyst)
    • define the process needed to handle the error condition

      • (at each subsystem boundary)
    • review the content, format, and wording of the error message
    • specify the manual procedures to correct errors
  9. Specify performance requirements: (Performance/Analyst)
    • check for conformity with performance and capacity requirements
    • determine specific performance or capacity measures for processes
  10. Specify data conversion: (Analyst with input from Development)
    • analyze data conversion requirements 
    • specify automated and manual data conversion process
    • specify conversion controls
  11. Specify Installation/Transition Requirements (Analyst with input from I&D).
  12. Specify transition from old to new Requirements (if applicable) (Analyst).
  13. Specify testing considerations, e.g., Integration Test, System Test, Client Acceptance, Installation/Transition (Analyst).
  14. Capture draft Requirements and ensure that traceability begins (Analyst).
Exit

2.4 Define Technical Operating Environment
Purpose  The Technical Operating Environment (TOE) is defined to document the development tools and supplier products (if applicable) so that team members know the target environment. This is the physical layout of the system.

 An entire TOE "document" is required for new projects. Extensively changing the physical environment requires updating or adding an addendum to existing "documentation." 

This activity occurs while developing Requirements.
Entrance 
Criteria 		 Baselined Thin Requirements 
Baselined Project Plan
Technology constraints or existing infrastructure 
Steps 
  1. Review current and planned hardware, software, and communications configuration

    2. (Development/System Administrator or CIS). 
  2. Define the proposed environment. Provide hardware, software and communications specifications allowing for growth (Development/System Administrator/CIS).

    3.  

     

    All applications should follow the Company guideline (Development/System Administrator/CIS) 

    Conduct Operational Readiness Review (ORR) including manageability. 
     

  3. Define physical development and testing environments which may include: (Development/System Administrator) 
    • hardware 
    • software 
    • development tools, e.g., editors, compilers, etc. 
    • testing tools 
    • SCM (Software Configuration Management) tools 
    • supplier-specific tools, e.g., Database Management system spreadsheet, etc. 
    Provide input to the Configuration Management Plan. See "Configuration Management" for details. 
    (Development/System Administrator, Architecture COE or Performance)
  4. Determine technical constraints to include the:
    • hardware
    • software
    • data communication links
    • target environment
    • specific hardware, software, and supplier products
    • data volumes
    • known response time limitations
    • known security vulnerabilities/limitations.
  5. Define system standards, which may include: (Development/System Administrator or Architecture COE)
    • programming standards (i.e., error message module or routines, debugging level, etc.)
    • backup and recovery procedures
  6. Define and establish performance logging/reporting requirements (Performance).
  7. Define and establish security requirements (e.g., warning banners, logging/audit) (Security).
  8. Produce TOE document (Development). 
  9. Conduct internal baseline review (Development/System Administrator or Architecture COE). See Baseline Process, Change Control Process, Document Control, and

    10. Reviews in the Appendix section for details. 
    The TOE document is now a controlled document under change control. 

    This step must be completed before Design the System begins. 

  10. Retain the appropriate artifacts in the Project Library for easy retrieval (Project Manager).

2.5 Develop Blueprint
Purpose  The Blueprint is the detailed plan for the development phase. The Blueprint is developed as a separate spreadsheet inventory of implementable units, modules, or CRs that are scheduled for a release. It is easier to manage if the blueprint is developed as a spreadsheet that is separate from the Project Plan. The Blueprint is an evolving entity. It may exist as early as the Project Plan, to include CRs in a release, and usually expands during the Define and Design phases.

 
Entrance 
Criteria 		 Baselined Thin Requirements
Baselined Project Plan 
Steps 
  1. Record implementable units, modules, scheduled CRs on the Blueprint or in the Project Plan (Development).
  2. Update the information to reflect completion of items (Development).

2.6 Develop Information Development Release Plan
Purpose  The Information Development Release Plan is developed to describe the anticipated Information Development needs for a project and to estimate the effort that will be required to deliver that support to the client. These needs may include, but are not limited to: the deliverable Information Development support needed for each target audience (such as on-line documentation, or paper documentation and training), the funding that should be made available for Information Development, and the schedule and strategy for development and delivery of Information Development deliverables.

 Developing the Information Development Release Plan is done at the same time as Requirements.

 This activity is required for new features, installation/transition, and new projects. The Project Manager and Information Developers jointly decide if this is required for CRs resulting from defects. 
Entrance 
Criteria 		 Baselined Thin Requirements
Baselined Project Plan 
Steps
  1. Develop an Information Development Release Plan that addresses the clients' education and training requirements, as well as the necessary strategy, resources, media, and schedule (Information Development). 
  2. Include in the Information Development Release Plan the strategy for client and other technical support documentation (if necessary) (Information Development). 
  3. Conduct internal baseline review (Information Development).

    4. See Baseline Process, Change Control Process, Document Control, and
    Reviews in the Appendix section for details. 
    The Information Development Release Plan is now a controlled document under change control. It should be reviewed at the regularly scheduled Project Status Meetings.

2.7 Develop Master Test Plan and Specifications
Purpose  The Master Test Plan is developed to ensure that the team understands the testing approach and strategy and the entrance and exit criteria for each testing phase. It is critical to clearly define the criteria needed to begin packaging if it is different than the exit criteria for System Test.

 The test cases and procedures (defined in the Test Case Specification) describe the actual test cases required for Integration and subsequent test phases. It is suggested that client, external interfaces, and their CIS counterparts are included in Test Plans and Test Case Specification development to gain maximum benefit of joint test planning (if practical).

 
Entrance 
Criteria 		 Draft TOE
Baselined Thin Requirements
Baselined Project Plan 
Steps 
  1. Identify resources to create Master Test Plan (Test/Development). 
  2. Create a new Test Case Specification document or update or add an addendum to an existing Test Case Specification document to define the test cases. Acceptable Test Case Specifications document use the template, or identify CR test case id's. 

    3.  

     

    Focus on the following types of tests: 

    • Acceptance: Verify that the system delivered to System Test meets the quality (entrance) criteria defined in the Test Plan. 
    • System (Project): Validate the system from a user workflow perspective including validation of Information Development material. 
    • Client Acceptance: Ensure that the test cases identified for customer acceptance execute successfully. 
    • Security: Validate data, benchmark, user identification and authorization, access control privileges, and other security requirements. 
    • Installation/Transition: Validate the installation/transition software from an information systems workflow perspective, including the validation of Installation/Transition documentation developed by the Information Development team or Project. 
    • Capability: Using live Operations System (OS) connections (simulators are not used), provide testing of multiple operation systems configured like the customer environment, and ensure that the business process flow functions across all impacted systems as defined in the Requirements. 
    • Performance: Verify that the system performs according to Requirements established, e.g., response time, throughput, etc. 
    • Other: Validate external interfaces, database conversion/upgrades and system start-up and shut-down, backup, recovery, batch runs, etc. 
    Specify 
    • input 
    • output (output values may include tolerances, expected features, such as update occurred, or other generic terms) 
    • other pertinent information

2.10 Baseline Requirements
Purpose  The Requirements are baselined so that any additional changes may be monitored under the Change Control process. As a result, a thorough and complete understanding is established between the client and the project team. Gaining this concurrence helps to maintain control over requirement changes, since these baselined requirements form the basis for the project's plans. 
Entrance Criteria  Updates to Baselined Thin Requirements
Baselined Project Plan 
Steps 
  1. Conduct internal review for all requirements

    2. (Analyst/Engineer).
    The team, the client, plus representatives from other affected systems, participate in the internal Requirements review to ensure that:
    • the Requirements accurately depict the client's business needs
    • the Requirements are accurate, thorough, and consistent
    • the client interface requirements support the client's tasks (if included in the Requirements document)
    • all the steps up to this point (e.g., specifying performance, security, control, backup, error handling requirements) are completed 
    • all requirements are testable
    See Baseline Process, Change Control Process, Document Control, and
    Reviews in the Appendix section for details. 

    The Requirements document is now a controlled document under change control. 

    Additionally, it is recommended that absolute numbering schemes be utilized to maintain traceability during change control.
     
     

  2. Obtain client concurrence verifying that the Requirements satisfy the client's stated business needs and meet commitments described in the Project Plan. Try to ensure that constituencies (e.g., internal CIS) are represented. Changes requested by the client are handled under change control (Analyst).
  3. Retain the appropriate artifacts in the Project Library for easy retrieval (Project Manager).

2.12 Conduct Requirements Lessons Learned
Purpose  A Lessons Learned Session is conducted after the Requirements are completed to evaluate the effectiveness of, and document improvements to, the Requirements process.

 This session is optional and may be initiated by any person involved in the phase. The person who initiates the Lessons Learned Session is responsible for appointing a facilitator.

 
Entrance 
Criteria 		 Baselined Requirements

 
Steps 
    1. Determine the participants. Schedule and announce the Lessons Learned Session (Initiator).
    2. Use brainstorming or other techniques to determine processes that were especially effective or could be improved

      3. (Facilitator).
    3. Write meeting minutes that include: (Facilitator)
      • action items
      • person responsible for action items
      • due date for action items. 
    4. Provide information to Project Manager to update the Quality Improvement Plan (QUIP) (if applicable). See Quality Improvement Plan (QUIP) (found in Appendix) for details (Initiator).

      5. Individual project teams may further refine these steps as necessary.
