6.1 Write Release Content Letter

Purpose  The Release Content Letter serves two purposes. The first is to tell the users what is being delivered and highlights database changes, performance impacts and restrictions. Secondly, it communicates to CIS what is to be installed and how. The CIS Front Door Process is also completed. 
Entrance Criteria  Pre-Release Letter with revised/updated System Content 
Steps 
  1. Write the Release Content Letter (Release Manager).
  2. Sign (and thus approve) the Release Content Letter

    3. (Project Manager).
  3. Complete CIS Q-2 review. 
  4. Retain the appropriate artifacts in the Project Library for easy retrieval (Project Manager).
Exit Criteria  Release Content Letter/Installation Guide
Approved Q-2 Review (Q-2)

6.2 Package the System for Deployment 
Purpose  The system is prepared and packaged for delivery/deployment to the client.

 
Entrance Criteria  Ship Sign-Off Checklist completed
All CRs are in an acceptable state for shipment, e.g., deferred, accepted, etc.
Release Content Letter 
Steps 
  1. Assemble and verify system, data, documentation and supporting material: 
    • Release Content Letter 
    • CRs in "ready to ship" state 
    • Supporting documentation as specified in the Information Delivery Plan 
    (Test, Installation and Deployment or Configuration Management (CM)) 
    This is the final test before shipping the system: Verify and document that the components developed, built and tested are the components packaged for the release (Integrity Report). 
    An example of the final test might be the client acceptance test(s), or a subset. 
  2. Perform final verification (e.g., SYSGEN, system Tables, component identifiers) (Installation and Deployment or CM). 
  3. Schedule delivery and initiate Implementation Request (IR) for CIS or equivalent. The IR has to be initiated and approved at least two weeks prior to implementation in production. IR approval satisfies CIS Project Deployment Process.
  4. Package the system and deliver to Deployment

    5. (Test, Installation and Deployment or CM). 
  5. Ensure the Commitment and Delivery Management sign-off is completed (Project Manager, Account Manager, Delivery Manager). 
Exit Criteria  Project delivered to Deployment
Approved Implementation Request
C&DM documents and criteria are met

Delivery Method sign-off

6.5 Perform Client Acceptance Test
Purpose  Client Acceptance Testing provides the opportunity to test and accept for production, new or extensively modified features of the project. Client Acceptance Testing is not intended to train clients on how to use the project. However, Information Development materials must be available to be used as part of the Client Acceptance Test. 

Each project is responsible for documenting the environment in which Client Acceptance Testing is performed. The criteria by which the client will accept the release is defined in the Client Acceptance Test Plan (CATP). 
Entrance Criteria System meets entrance criteria (defined in the Master Test Plan)
Client Acceptance Test Plan 
Steps
  1. Make room, equipment, etc. arrangements. 
  2. Prepare and verify Client Acceptance Test environment and data (Test). 
  3. Conduct Client Acceptance Test sessions (Test). 
  4. Document Client Acceptance Test outcome (e.g., CRs entered or more formal documentation) (Test). 
  5. Solicit feedback from clients on aspects of the session that were especially effective or could be improved (Test). 
  6. Retain appropriate artifacts in the Project Library for easy retrieval (Project Manager). 
  7. Client sign-off. 
Exit Criteria Client Acceptance Test completed (meets Master Test Plan exit criteria) (Q-2)
CRs entered (if applicable) 
Quality Records Client Acceptance sign off 
Client Acceptance Test Faults (logged in problem tracking tool)

6.6 Conduct Installation and Deployment Lessons Learned Session 
Purpose  An Installation and Deployment Lessons Learned Session may be held after the system is deployed or declared Production Ready (PR) to evaluate its effectiveness and document improvements.

 This session is optional and may be initiated by any person involved in this phase. The person who initiates the Lessons Learned Session is responsible for appointing a facilitator.

 
Entrance Criteria  Project installed at client site or declared PR 
Steps 
  1. Determine participants, schedule, and announce the Lessons Learned Session. (Initiator)
  2. Use brainstorming or other techniques to determine processes that were especially effective or could be improved (Facilitator).
  3. Write meeting minutes that include: (Facilitator)
    • action items 
    • person responsible for action items
    • due date for action items
    (Facilitator)
  4. Provide information to Project Manager to update the Quality Improvement Plan (QUIP) (if applicable). See Quality Improvement Plan (QUIP) for details. (Initiator)
Individual project teams may further refine these steps as necessary.

 
Exit Criteria  Meeting minutes and/or Action Item Register 
Quality Records  QUIP
Meeting minutes and/or completed Action Item Register 
Controlled Documents  None 
Metrics  None 
Templates  None 
Tools  As Appropriate

6.7 Close the Project 
Purpose  The project is closed to bring an official end to the project. 
Entrance Criteria  Project declared PR
Formal Lessons Learned conducted 
Steps 
  1. Conduct project closure by: (Project Manager) 
    • closing all CRs that have been delivered with the release (CRs should be closed within one week of PR)

6.8 Conduct Formal Lessons Learned Session 
Purpose  The formal Lessons Learned Session is periodically conducted to evaluate the effectiveness of the completed life cycle processes and to recommend improvements to these processes, including the QMO. 

The formal Lessons Learned Session is the only Lessons Learned Session that is required. The Project Manager is responsible for this Lessons Learned Session and for appointing a facilitator. 

The Lessons Learned Session must cover costs and schedules as well as the technical aspects of the project to evaluate the effectiveness of the entire process and document necessary improvements as appropriate. A project may choose to include clients in addition to project team members in the session. 
Entrance Criteria  Project declared PR
Baselined Project Schedule and Costs
Last Planned Project Schedule and Costs
Actual Project Schedule and Costs 
Steps 
  1. Determine the participants, schedule, and announce the Lessons Learned Session 

    2. (Project Manager). 
  2. Perform an initial analysis and document the differences between the original, baselined, last planned, and actual project schedule and costs (Project Manager).

    3. The output may be a report or a series of viewgraphs to be presented during the Lessons Learned Session. 
  3. Use brainstorming or other techniques to determine processes that were especially effective or could be improved (Facilitator). 
  4. Write meeting minutes that include: (Facilitator)
    • action items
    • person responsible for action items
    • due date for action items. 
  5. Update the Quality Improvement Plan (QUIP). See Quality Improvement Plan (QUIP) in the Appendix for details. This session may also initiate changes to the project's local procedures or a change request for the QMO itself (Project Manager). 
Exit Criteria  Meeting Minutes and/or Action Item Register
Updates to the QUIP, local procedures 
Change request for QMO (if applicable)
Updates to Project Closeout documentation that covers Lessons Learned, Cost and Schedule Variances, and the final Project metrics 
Quality Records  Updated QUIP 
Meeting minutes and/or completed Action Item Register
IT Assessment 
Controlled Documents  Final version of schedule
Copy of risks managed/tracked throughout the project

6.9 Collect On-going Client Satisfaction Feedback on the Project 
Purpose  On-going, evaluative feedback on the project is solicited, at a minimum, annually. This is done to ensure that we are meeting clients' needs. Client feedback helps in assessing their ability to transfer information to the job and determine the ongoing value of the project. 
Entrance Criteria  Project delivered to the client

 
Steps 
  1. Identify clients to be solicited for feedback 
  2. Solicit and collect feedback. 
  3. Assess feedback. 
  4. Where appropriate, use feedback to improve the process or project. 
Exit Criteria  Feedback solicited

 
Quality Records  Assessment available on IT-PTS
