3.1 Document System Architecture
Purpose  The System Architecture defines/documents the logical view of the system. The entire System Architecture is required for new projects. Extensively changed systems or new feature/CR work that impacts the architecture requires updating or adding an addendum to existing documentation.

 The system specified within the System Architecture is:
 
 

  • reusable
  • maintainable
  • modular
  • identifies common routines
  • ensures that the requirements can be met 
  • follows Company standards 
Entrance Criteria  TOE
Baselined Requirements 
Steps  This section overlaps with Section 2.1 - Define the Project
  1. Capture the logical view of the system or changes to the architecture (Development). 
    2. Consider:
    • An overview of the system, identifying the mission of the system and design objectives set forth in order to fulfill this mission
    • Assumptions or constraints
    • Strategies for meeting usability objectives
    • Performance requirements
    • Security requirements
    • Hardware needs
    • Software needs
    • Subsystems and functions/process control
    • Major internal and external interface relationships
    • Common routines
    • Database considerations
    • Outstanding issues and/or future considerations
    • Installation, growth, and retrofit considerations.
    Use the Pre-Commitment Architecture Specification as a source of information. 
  2. Conduct internal review of Draft System Architecture (Development).

    3. See Baseline Process, Change Control Process, Document Control, and Reviews in the Appendix section for details.
  3. Retain the appropriate artifacts in the Project Library for easy retrieval (Project Manager).
Exit Criteria  Baselined System Architecture (Q-9)

3.2 Identify Reusable Code
Purpose  Reusable code (modules that may be reused by this system or modules within the system that are candidates for future reuse) is identified to save resources and usually results in cost savings for the client.

TOE

  2. Identify potential modules and determine the feasibility for reuse (Development).

3.3 Evaluate and Acquire Supplier Software Product 
Purpose  Supplier products are acquired and evaluated to ensure minimum quality standards and that specific product requirements are met. 

Reference Approved Product List (APL) change and waiver process. 
Entrance Criteria  Individual projects, based on client requirements and development needs, make a decision to use supplier product(s).

 
Steps 
  1. Identify critical suppliers, based on their impact to Company's success. Products which have been previously approved will be listed on the APL. 
  2. If a project selects a product which has not been previously approved for use on the APL, then the project must follow the APL waiver process. 
  3. Follow Software Procurement. 
Exit Criteria  Supplier products approved; with a granted APL waiver, on the approved APL, added to the APL or approved as part of the APL exception process. 
Quality Records  A document describing how the product was evaluated for selection should be retained by the originating organization.

3.4 Design the Database 
Purpose  Designing the database is necessary to produce a physical database design.

 A complete Database Design is required for new projects. Extensively changing the database requires updating or adding an addendum to existing documentation. 

Other changes are documented in the Detailed Design.

 
Entrance Criteria  TOE
Baselined Requirements
System Architecture document (draft) 
Steps  This activity usually overlaps with Section 2.1 - Define the Project.
  1. Develop database designs as needed to: (Development/Data Modelers/ DBA)
    • Define logical and physical model and characteristics such as: 
      • mapping 
      • data types 
      • indices 
      • relationship (E-R diagrams) 
      • map records 
    • Specify options
    • Calculate space/capacity
    • Evaluate performance (Development or Performance)
    • Evaluate security (Development or Security)
  2. Design Installation/Transition Procedures (Development, I & D).
  3. Design the conversion routines (data, environmental, optional) (if applicable) (Development).

    4. See Baseline Process, Change Control Process, Document Control, and
    Reviews in the Appendix section for details. 
    The Database Design is now a controlled document under change control. 
  4. Retain the appropriate artifacts in the Project Library for easy retrieval (Project Manager). 
Exit Criteria  Baselined Database Design (Q-9)
Baselined Installation/Transition Design
Conversion Routines (if applicable)

3.5 Develop Detailed Design 
Purpose  The Detailed Design is developed to describe how the system will be built, as well as what tools and what techniques will be used to build it.

 This activity is required for a new project. It is also required for any new feature, a CR that contains or will recall reusable code, is complex, or being significantly changed. For new features that do not meet this criteria, the design information may be documented within the Requirements. CRs that do not meet this criteria must have the design documented within the CR record in the trouble tracking system.

 
Entrance Criteria  Baselined Project Plan
Baselined Requirements
TOE
System Architecture (draft) 
Steps  This section overlaps with section 2.1 - Define the Project. 
  1. Select design strategy if multiple options are mentioned in system standards (e.g., object oriented, structured, etc.) (Development).
  2. Decompose required functionality into implementable units (Development). 
  3. Evaluate performance (Performance or Development).
  4. Evaluate security (Security or Development).
  5. Evaluate Installation/Transition impacts (I & D). 
  6. Design implementable software units per the Detailed Design Template (Development).
  7. Conduct internal baseline review after the Requirements have been baselined (Development).

    8. See Baseline Process, Change Control Process, Document Control, and
    Reviews in the Appendix section for details. 
    The Detailed Design is now a controlled document under change control. 
  8. Retain the appropriate artifacts in the Project Library for easy retrieval (Project Manager).
  9. Review and update estimates. 
  10. Determine commitment dates. 
Exit Criteria  Baselined Detailed Design (Q-9)
Commitment date for Q-2 (Q-9)
Decision to proceed to Q-2 (Q-9)
