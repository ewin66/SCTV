5.1 Perform System Test
Purpose  System Test is performed by assembling the modules, data, installation/transition software and Information Development materials into a project and validating that the system meets the client needs as stated in the Requirements.

 
Entrance Criteria Integration Test Sign-Off Criteria Sheet
Meets System Test Entrance Criteria defined in Master Test Plan
Updated Information Development Materials (Installation, Transition and On-line Materials from Integration Test)
Documentation and Training Material as defined in Information Development Release Plan 
Steps 
  1. Execute test cases defined in the Test Case Specifications (Test) 
    • Acceptance: Verify that the system delivered to System Test meets the quality (entrance) criteria defined in Master Test Plan/Test Case Specifications. 
    • System (Project or Feature): Validate the system from a client workflow perspective including validation of Information Development material. 
    • Customer Acceptance: Ensure that the Test Case Specifications test cases identified for the client acceptance execute successfully. 
    • Security: Validate data, benchmark, client identification and authorization, access control privileges, and other security requirements. 
    • Installation/Transition: Validate the installation/transition software from an information systems workflow perspective, including the validation of Installation/Transition documentation developed by Information Development or the Project. 
    • Capability: Using live Operating System (OS) connections (simulators are not used), configured like the client environment, ensure that the business process flow functions across all impacted systems as defined in the Requirements. 
    • Other: Validate external interfaces, database conversions/upgrades and system start-up and shut-down, backup, recovery, batch runs, etc. 
    Activities include: 
    • Identify tests to be executed. 
    • Execute the test steps. 
    • Document any test discrepancies using a problem tracking system. 
    • Document results. 
    • Review actual and expected results; there should be no discrepancies. 
    • Ensure proper testing is performed once a fix is made by direct traceability from discrepancy reports to test case (use the problem tracking system). 
    • Test the Information Development materials for overall accuracy. For the purpose of establishing metrics and ensuring traceability, the use of an on-line problem tracking system is preferred to Review Tracking Sheets during System Test. 
    • Retain or reproduce successful (passed) test results for 48 hours. 
  2. Perform regression tests (Test). 
  3. Maintain test metrics information for test tracking (Test).

    4. The weekly summary of System Test metrics are transient quality records; that is, the new metrics will supersede the old. Only the final week summary of System Test metrics will be maintained as the quality record after successfully passing System Test. 
  4. Decide if the test metrics and System Test results support decision to ship. If the decision is "yes", then complete the Ship Sign-off Checklist. If the decision is "no", notify the Project Manager (Test). 
  5. Sign the Ship Sign-Off Checklist verifying the system is ready to be shipped (System Test Manager). The signature of the Development Senior Director signifies that the release meets the contractual agreements negotiated with the client and that third party software licenses (if any) have been procured as necessary (Senior Development Director). The Senior Director of Development may designate another signee for a pure maintenance release (containing no features, minor or major). 
  6. Update automated test scripts (if applicable) (Test). 
  7. Retain the appropriate artifacts in the Project Library for easy retrieval (System Manager). 
  8. Perform Operational Readiness Lab (ORL) testing if required for your project. These tests are designed to ensure the application is ready for introduction into the production environment. Test cases are executed in a production-like environment and include installation, performance and interoperability with other applications. These test may be performed by the System Test group or by the CIS Network/ORL Group. 
Exit Criteria Ship Sign-off Checklist (met System Test exit criteria defined in Master Test Plan) (Q-5) 
Test metrics (final week summary only) (Q-5)
Updated test scripts (if applicable) (Q-5) 
Quality Records Ship Sign-off Checklist
Test metrics (final week of System Test)
Final pass/fail status of each System Test case including Client Acceptance Test Case(s) 
System Test faults (from problem tracking tool)
ORL test results report (if required) 
Controlled Documents None 
Metrics Test metrics
Updates to or actuals for size and cost 
Templates Ship Sign-off Checklist
Tools For tool recommendations visit SCM Tools Group

5.2 Conduct Performance Validation 
Purpose  Performance Validation demonstrates that the system meets or exceeds a set of specific performance requirements. This validation could include capacity, response times, throughput, availability and message sizes, based on the specific requirements indicated to be pertinent to the project in the performance requirements.

 Performance validation may be done using project benchmark suites, project test cases, analytical models, simulation models, sampling techniques, or other techniques based on a combination of practicality, efficiency and accuracy as determined by the performance analysts and the project.

 
Entrance Criteria  Baselined Performance Requirements indicating specific performance criteria,
environment, and code 
Steps  If any of the areas (capacity, response time, throughput, availability, and message sizes) have quantified performance requirements:
 
 
  1. Create and document a Performance Validation Plan:
    • Construct performance specific test cases or benchmark suites to be measured or identify test cases in the Test Case Specifications that will be used to vaildate the objectives in the performance requirements.
    • Identify target measurement environment(s) (Performance and/or System Test).
    • Identify one or a combination of measurement techniques and tools to execute performance tests and/or analyze results (e.g., stop watch, internal processing time statistics, throughput rates, analytical modeling, simulation modeling sampling techniques, investigation of hardware mean time between failures, verification of on-line software upgrade capabilities, verification of conversion and transition duration, etc.)

      • (Performance and/or System Test).
    • Document Performance Validation Plan (Performance and/or System Test).
  2. Conduct internal baseline review of the draft Performance Validation Plan (Performance and/or System Test).

    3. See Baseline Process, Change Control Process, Document Control, and
    Reviews in the Appendix section for details. 
    The Performance Validation Plan is now a controlled document under change control. 
  3. Retain the appropriate artifacts in the Project Library for easy retrieval (Project Manager). 
  4. Execute Performance Validation Plan and document results: 
    • Establish target measurement environment(s) (Performance and/or System Test and/or Environment Support).
    • Execute test cases and obtain measurements (Performance and/or System Test).
    • Analyze measurements and document results (Performance).
Exit Criteria  Baselined Performance Validation Plan
Performance measurements obtained and analyzed
Performance results documented, including areas requiring improvement, if needed

Completed Action Item Register (if applicable) 
Meeting minutes (if applicable)

5.3 Perform Guest Test (if applicable)
Purpose  Guest Testing (client Feedback Testing) provides clients with the opportunity to test new or extensively modified features of the project. Guest Testing is not intended to train clients on how to use the project. However, Information Development materials that are available should be used wherever possible so that early feedback can be obtained.

 Guest Testing is optional depending on how much of the project is new or changed and if the cost of Guest Testing is justified. The Test organization makes this decision and documents it in the Master Test Plan.

 Each project is responsible for documenting when Guest Test is performed, and for defining and meeting the entrance and exit criteria.

 
Entrance Criteria  Decision to perform Guest Testing from Test Plan
System meets entrance criteria (defined by project)

 
Steps
  1. Determine Guest Test objectives (Test).
  2. Make arrangements for room, equipment, etc (Test).
  3. Prepare and verify Guest Test environment and data (Test).
  4. Conduct Guest Test session (Test).
  5. Document Guest Test outcome (e.g., CRs entered or more formal documentation) (Test).
  6. Solicit feedback from clients on aspects of the session that were especially effective or could be improved (Test).
  7. Retain appropriate artifacts in the Project Library for easy retrieval (Project Manager). 
Exit Criteria  Guest Test completed (meets project documented exit criteria)
CRs entered (if applicable)

 
Quality Records Guest Test outcome
Guest Test faults (from problem-tracking tool) 
Controlled Documents  None 
Metrics  Test metrics (only if directly related to Guest Test)
Updates to or actuals for size and cost 
Templates  None

5.5 Conduct System Test Lessons Learned Session 
Purpose  A Lessons Learned Session is conducted after System Test is completed to evaluate the effectiveness of, and document improvements to, the Testing process.

 This session is optional and may be initiated by any person involved in the phase. The person who initiates the Lessons Learned Session is responsible for appointing a facilitator.

 
Entrance Criteria  Completion of System Test 
Steps 
  1. Determine the participants. Schedule and announce the Lessons Learned Session (Initiator).
  2. Use brainstorming or other techniques to determine processes that were especially effective or could be improved (Facilitator).
  3. Write meeting minutes that include: (Facilitator)
    • action items
    • person responsible for action items
    • due date for action items 
  4. Provide information to Project Manager to update the Quality Improvement Plan (QUIP) (if applicable). See Quality Improvement Plan (QUIP) (found in Appendix) for details (Initiator).
Individual project teams may further refine these steps as necessary.

 
Exit Criteria  Meeting minutes and/or Action Item Register 
Quality Records  QUIP
Meeting minutes and/or completed Action Item Register

 
Controlled Documents  None 
Metrics  None 
Templates  None
