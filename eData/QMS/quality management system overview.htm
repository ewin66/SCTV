Quality Management System Overview


Purpose and Scope

The purpose of instituting effective software processes at Company IT is to measurably improve productivity, quality of the delivered project, and predictability of project cost and schedule. The approach to the quality system is through three tiers.

The top tier, Tier 1, is enforced by Software Development management and describes in a few sentences the mission and vision of the organization.

The middle tier, Tier 2, is the thin, common sense, standard methodology applicable to all projects. It is referred to as the Quality Method of Operation (QMO). It is a process architecture, not prescriptive, that describes the "what" of software projects and not the "how." As best practices are identified and adopted, they will be added to the QMO to more clearly define the common "what."

The lower tier, Tier 3, is the collection of local practices "how" the project proceeds to accomplish the methodology outlined in Tier 2. Tier 3 assembles those practices that work within the COEs and propagates them to other projects.

The entire system is called the Quality Management System (QMS). Following the QMO:

  • emphasizes client involvement throughout the project
  • stresses building in quality up-front
  • specifies accountability
  • defines entrance and exit criteria for each phase
  • provides templates for key outputs
  • employs teaming as a key aspect of project management
  • embodies a continuous process improvement philosophy
Compliance Requirements

All projects must comply with the QMO. The point in time of compliance depends on where in the software development life cycle a project is. Compliance should be on a going-forward basis. At a minimum, all software projects must be planned and managed as projects, and the QMO entrance and exit criteria must be met. To comply, the project must be able to demonstrate to a QMO auditor that the project has performed the activities and steps defined in the QMO.

All projects upon adoption of the QMS must develop and implement a plan to develop their tier 3 documentation. This plan is the project's first QUIP.

An audit schedule will be followed with emphasis on those projects that have not yet demonstrated compliance. Audit results are reported following the QMO audit procedures. Once a project demonstrates compliance, audits will become less frequent.

Demonstration of compliance is provided by the project through:

  • controlled documents and quality records, as required by the QMO
  • interviews by the audit team
Following the QMO will ensure a project is "Operating At" CMM Level 2.
QMO Framework Structure

The QMO consists of 7 phases, an Introduction, and an Appendix:

  1. Initiate the Project
  2. Define the Project
  3. Design the System
  4. Build the System
  5. Test the System
  6. Deploy the System
  7. Support the System
Each QMO phase is structured as follows:
  • Purpose: explains the reason an activity is performed
  • Entrance Criteria: events that must be completed to begin a phase
  • Steps: a list of actions that are performed in order to produce a quality project
  • Exit Criteria: events that must have occurred to complete the phase
  • Quality Records: an artifact that demonstrates the project is following the QMO. These are typically
    • Meeting Minutes: summary notes of meeting highlights and decisions made.
    • Action Item Registers: logs of action items, and their status. It may also contain risks, issues, or other pertinent items to be tracked.
    • Review Tracking Sheets: contain electronic or "ink" signatures used to demonstrate that stakeholders have agreed to a controlled document.
  • Controlled Documents: an artifact that is under change control, meaning there are version numbers
  • Metrics: simple metrics to be collected to be used once CMM Level 2 has been achieved
  • Templates: provided for key outputs, like Requirements, Project Plans, Master Test Plans, Test Case Specifications, etc.
  • Tools and Techniques: includes aids that a project can use to perform the steps within an activity
QMO is a process architecture and may be implemented as a waterfall model, relaxed waterfall, spiral, incremental, iterative, etc. All stages must be complete; however, activities in each phase may overlap with activities from other phases. All that is required is that the entrance and exit criteria have been met, and that all quality records, controlled documents, and metrics are retained in a common repository.

A phase is not complete until all of the Exit Criteria have been completed.
